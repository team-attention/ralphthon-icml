# experiments.jsonl Template

For a new campaign, initialize it as an empty file with `touch experiments.jsonl`, then confirm it has no lines. If the file already exists, do not truncate or replace it. Write one compact JSON object per completed run.

The object below is a schema example only. **Do not copy** it into the ledger; it is not evidence and is deliberately not a placeholder baseline:

```json
{"run_tag":"example-only","trial":"candidate-1","git_sha":"1111111111111111111111111111111111111111","remote_commit":"1111111111111111111111111111111111111111","hypothesis":"one documented hypothesis","change":"one train.py change","val_bpb":1.2345,"peak_vram_mb":2048.0,"num_params_M":50.3,"elapsed_seconds":325.0,"status":"discard","failure":null,"next_hint":"try a different single change","wandb_run":"offline-example","cookbook_sha":"97a0af14b0acae042162b1f70f17fbe2d570afa2","vessl_job_slug":"autoresearch-example-abc123","vessl_job_name":"autoresearch-example-1111111","vessl_job_state":"succeeded","approved_resource_spec":"resourcespec-live-a100","job_resource_spec":"resourcespec-live-a100","gpu_identity":"NVIDIA A100-SXM4-80GB","gpu_count":1,"branch":"autoresearch/example-only","cache_fingerprint":"sha256:aaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaa","evaluation_fingerprint":"sha256:bbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbb","train_py_sha256":"sha256:cccccccccccccccccccccccccccccccccccccccccccccccccccccccccccccccc","log_sha256":"sha256:dddddddddddddddddddddddddddddddddddddddddddddddddddddddddddddddd","job_json_sha256":"sha256:eeeeeeeeeeeeeeeeeeeeeeeeeeeeeeeeeeeeeeeeeeeeeeeeeeeeeeeeeeeeeeee"}
```

`run_tag` identifies the campaign and `trial` identifies one run within it. The cookbook SHA, separately supplied local/remote full commits, unique VESSL Job slug/name/terminal state, approved and actual resource specs, structured A100 model/count, remote branch, canonical cache hash, evaluation hash, `train.py` hash, local log hash, and saved Job-JSON hash correlate the local ledger with remote execution. The W&B allowlist includes those correlation values plus approved metrics; raw paths/files and ledger-only hypothesis, change, failure, and next hint are not uploaded as W&B config or metrics.

Allowed status values are `keep`, `discard`, `crash`, and `confirmation`. Baseline is keep/crash; candidates are keep/discard/crash; winner-confirmation is confirmation/crash and requires a prior kept candidate. Record baseline first, candidates sequentially without duplicates, and confirmation last. A kept candidate must have strictly lower `val_bpb` than the current best. Confirmation uses the unique `autoresearch/<run-tag>-confirm` branch but the same full commit, hypothesis/change, and `train.py` hash as the best kept candidate, and a successful confirmation must remain below baseline. Resource, GPU, cache, evaluation, and cookbook identities stay fixed; Job names/slugs and raw-evidence hashes stay unique. A crash requires a nonempty `failure` and a failed, terminated, or cancelled Job. Never add credentials, private participant data, console logs, dataset contents, checkpoint contents, or artifacts. The recorder uses an exclusive ledger lock, one-line append, `fsync`, and rollback; never rewrite or remove an earlier line manually.
